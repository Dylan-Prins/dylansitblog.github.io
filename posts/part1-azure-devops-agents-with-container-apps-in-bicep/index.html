Part 1: Azure DevOps agents with Container apps in bicep  [draft]

February 22, 2022 · 4 min · Me | Suggest Changes

In my journy to find the best environment for Azure DevOps agents I have found (in my opinion) the best solution there is for now. I have tried multiple solutions for hosting the agents and I will explain the cons and pro’s of the tested solutions.

Solution Pro’s Cons
VM Scale set Fast, Only option when building containers Expensive
Container Instances Very cheap Not easy to scale
Kubernetes service Very scalable by using KEDA Expensive

And then Container apps came. Container apps let you deploy containers in a Microsoft managed namespace that is running on Kubernetes. This has the advantage that you can use KEDA (Kubernetes event-driven autoscaling) to perform the scaling. In this guide we will use KEDA to check Azure DevOps for Jobs in de queue. When there are more jobs waiting, KEDA wil create new containers and add them to the pool.

What we will create in this guide:

  • Azure Container Registry
  • Azure DevOps container
  • Log Analytics Workspace
  • Kube Environment (for hosting the container app)
  • Container App

In the guide I will use Azure CLI to deploy the bicep templates.

Part 1: Setting up the DevOps agent#

Azure Container Registry#

First we need to create an Azure container registry, so that we can host the devops agent there.

DevopsAgent\template.bicep

param location string = 'westeurope'

resource ContainerRegistry 'Microsoft.ContainerRegistry/registries@2021-12-01-preview' = {
  name: 'containerregistryieufhwibfw'
  location: location
  properties: {
    adminUserEnabled: false
    anonymousPullEnabled: false
    dataEndpointEnabled: false
  }
  sku: {
    name: 'Basic'
  }
}

Deploy the template with the following commands:

az deployment group create --name containerapps --resource-group [resource group] --template-file .\DevOpsAgent\template.bicep

Build de Azure DevOps container#

Create the 2 files below. These files are almost a copy from the Microsoft documentation page: Run a self-hosted agent in Docker. Make sure that the files are created in the same directory. The docker file needs the start.sh script when building the agent.

DevOpsAgent\DockerFile

FROM ubuntu:18.04

# Update the ubuntu image and install apt-transport-https
RUN apt-get update -y && apt-get install -y apt-transport-https

# Install agent requirements and clean up
RUN apt-get install -y --no-install-recommends \
  ca-certificates \
  curl \
  jq \
  git \
  iputils-ping \
  libcurl4 \
  libicu60 \
  libunwind8 \
  netcat \
  libssl1.0 \
  && rm -rf /var/lib/apt/lists/*

# Install Azure CLI and clean up
RUN curl -LsS https://aka.ms/InstallAzureCLIDeb | bash \
  && rm -rf /var/lib/apt/lists/*

# Can be 'linux-x64', 'linux-arm64', 'linux-arm', 'rhel.6-x64'.
ENV TARGETARCH=linux-x64

WORKDIR /azp

COPY ./start.sh .
RUN chmod +x start.sh

ENTRYPOINT ["./start.sh"]

DevOpsAgent\start.sh

#!/bin/bash
set -e

if [ -z "$AZP_URL" ]; then
  echo 1>&2 "error: missing AZP_URL environment variable"
  exit 1
fi

if [ -z "$AZP_TOKEN_FILE" ]; then
  if [ -z "$AZP_TOKEN" ]; then
    echo 1>&2 "error: missing AZP_TOKEN environment variable"
    exit 1
  fi

  AZP_TOKEN_FILE=/azp/.token
  echo -n $AZP_TOKEN > "$AZP_TOKEN_FILE"
fi

unset AZP_TOKEN

if [ -n "$AZP_WORK" ]; then
  mkdir -p "$AZP_WORK"
fi

export AGENT_ALLOW_RUNASROOT="1"

cleanup() {
  if [ -e config.sh ]; then
    print_header "Cleanup. Removing Azure Pipelines agent..."

    # If the agent has some running jobs, the configuration removal process will fail.
    # So, give it some time to finish the job.
    while true; do
      ./config.sh remove --unattended --auth PAT --token $(cat "$AZP_TOKEN_FILE") && break

      echo "Retrying in 30 seconds..."
      sleep 30
    done
  fi
}

print_header() {
  lightcyan='\033[1;36m'
  nocolor='\033[0m'
  echo -e "${lightcyan}$1${nocolor}"
}

# Let the agent ignore the token env variables
export VSO_AGENT_IGNORE=AZP_TOKEN,AZP_TOKEN_FILE

print_header "1. Determining matching Azure Pipelines agent..."

AZP_AGENT_PACKAGES=$(curl -LsS \
    -u user:$(cat "$AZP_TOKEN_FILE") \
    -H 'Accept:application/json;' \
    "$AZP_URL/_apis/distributedtask/packages/agent?platform=$TARGETARCH&top=1")

AZP_AGENT_PACKAGE_LATEST_URL=$(echo "$AZP_AGENT_PACKAGES" | jq -r '.value[0].downloadUrl')

if [ -z "$AZP_AGENT_PACKAGE_LATEST_URL" -o "$AZP_AGENT_PACKAGE_LATEST_URL" == "null" ]; then
  echo 1>&2 "error: could not determine a matching Azure Pipelines agent"
  echo 1>&2 "check that account '$AZP_URL' is correct and the token is valid for that account"
  exit 1
fi

print_header "2. Downloading and extracting Azure Pipelines agent..."

curl -LsS $AZP_AGENT_PACKAGE_LATEST_URL | tar -xz & wait $!

source ./env.sh

print_header "3. Configuring Azure Pipelines agent..."

./config.sh --unattended \
  --agent "${AZP_AGENT_NAME:-$(hostname)}" \
  --url "$AZP_URL" \
  --auth PAT \
  --token $(cat "$AZP_TOKEN_FILE") \
  --pool "${AZP_POOL:-Default}" \
  --work "${AZP_WORK:-_work}" \
  --replace \
  --acceptTeeEula & wait $!

print_header "4. Running Azure Pipelines agent..."

trap 'cleanup; exit 0' EXIT
trap 'cleanup; exit 130' INT
trap 'cleanup; exit 143' TERM

chmod +x ./run-docker.sh

# To be aware of TERM and INT signals call run.sh
# Running it with the --once flag at the end will shut down the agent after the build is executed
./run-docker.sh "$@" & wait $!

Now we can build the Azure DevOps agent into a Docker container. Go to the folder containing the the files and start the build. It is important to begin with the URL of the ACR followed by the container name. Otherwise we can not push it to the ACR. I always use SemVer versioning, which is why I start with version 1.0.0.

docker build -t containerregistryieufhwibfw.azurecr.io/dockeragent:1.0.0 .

Log in to azure with azure CLI and then authenticate with the ACR

az login

az acr login -n containerregistryieufhwibfw

Let’s finally push the container to the container registry:

docker push containerregistryieufhwibfw.azurecr.io/dockeragent:1.0.0

You should now have an Azure container registry container that is hosting an Azure DevOps container image. in the next part we will run an Azure container app with this image
